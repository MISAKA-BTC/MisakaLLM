## §20.4 発行の配管(25/5 に改訂)
`FeeSplitParams` の service 系フィールド(borsh 互換のため 0 で温存中)を compute pool として復活させる。**フィールド追加ゼロ**。

| | 現行(DNS Active) | HF-MIL後(提案A・推奨) |
|---|---|---|
| subsidy worker(base + inclusion) | 62 + 8 = **70%** | **70%(不変)** |
| subsidy validator(§E) | **30%** | 名目 **25%**(実効 25〜30%、滑走。25 = 引上げ前の原点水準がフロア) |
| subsidy compute(service復活) | 0% | **5%**(上限10%は Phase C = TNS発効を条件とする HF) |
| normal fee / finality fee | 90/10/0・25/75/0 | normal不変。finality feeは Phase Bで **25/65/10** へ(compute枠) |

25/5 の根拠: 25→30 引上げが「stakeファイナリティ強化」のために足した +5pp をそのまま第2アテスタ集合の購入に充て、既存バリデータは原契約水準(25%)を決して割らない — ガバナンス上これ以上防御しやすい分割線はない。Phase 進行に伴う報酬追従は subsidy でなく手数料側で行う: Phase B で ComputeDepth が MIL 大口決済をゲートし始めた時点で、finality fee を 25/75/0 → 25/65/10 に開ける。§20.7 実装タッチポイント: `dns_finality.rs:2093`(FeeSplitParams)= subsidy validator 2500 / compute 500 設定。

## §23 マルチアセット決済とMSKバイバック — 公式ペイメントゲートウェイ

### 23.1 原則: プロトコルはMSKしか知らない
USDT / USDC / SOL / ETHの受け入れは、エスクロー・レシート・バーン・88/5/4/3分配のどこにも入れない。多通貨化はエッジ — §14.3のpaymasterを正式化した**公式ペイメントゲートウェイ**(運営主体のオフプロトコルサービス)— の会計方針として実装する。理由: (a) 多通貨エスクローは契約・監査面を倍加させる、(b) ブリッジリスクを決済層に持ち込まない、(c) §6.1の単一トークン原則とバーン機構を無傷に保つ。プロバイダが受け取るのは常にMSKであり、ネットワークから見ればゲートウェイは「大口のMSK支払いユーザー」に過ぎない。**凍結条項**: (i) プロバイダは非MSKアセットにプロトコル経由で一切触れない(為替・鍵管理・税務・規制当事者性をプロバイダから遮蔽し、PQ環境の外に出さない)。(ii) 運営は逆方向の換金代行(MSK → USDT/USDC/SOL/ETH等)を**提供しない** — 順方向はクレジット販売 + 自己勘定トレジャリ方針として整理できるが、逆方向の提供は暗号資産交換業の射程に直結し、本節の法的構造の生命線を切る(§23.5)。MSKの現金化は各保有者が市場で行う。

### 23.2 アーキテクチャ — 在庫モデル
```
入金: Ethereum(USDT/USDC/ETH)・Solana(USDC/SOL)
      ユーザー毎デポジットアドレス、閾値署名(MPC)カストディ
      チェーン毎ファイナリティ確認後にクレジット発行
        ↓
クレジット: MSK建て・アカウント束縛・非譲渡・原則非払戻
        ↓
消費: ゲートウェイが自己のMSK在庫からセッションエスクローをopen(§14.3経路)
        ↓
補充: 入金純額の90%をTWAPバッチでMSK市場買い → 在庫へ
      10% → 運営ウォレット(プロトコルTreasury〔手数料3%〕とは別会計)
```
**アトミックスワップにしない**ことが要点。ユーザー体験(即時クレジット・即時利用)を薄い板での執行から切り離し、買付はスリッページ上限(例: バッチあたり2%)付きで時間分散する。上限超過時は翌窓へ繰り延べ、ボラティリティ急騰時は見積りスプレッドを動的に上乗せする。

### 23.3 90/10の配管と価格式
非MSK払いの見積り(一般形): `P = ask_MSK × R_oracle × (1 + b) ÷ 0.9`(R = FSL price fact参照のMSK/USDレート、セッションまたはトップアップ単位でロック。b = **バーンマージン率**、公開パラメータ)。支払額Pのうち 0.9P が全額MSK買付に回り、その内訳が原価(COGS = ask×R相当、在庫へ)と余剰(b×ask×R相当、**即時バーン**)に分かれ、0.1P が運営収入となる。b = 0でv0.8の式に退化しバーンはゼロ — **バーンは価格マージンからしか生まれない**という会計恒等式(§23.8)を価格式自体に露出させる。系として**MSK直払いが常に最安**(スプレッド0)— 追加のMSK保有インセンティブ。クレジットをMSK建てにするのは運営の為替リスクを消すため: 買付執行価格でクレジット化すれば、負債(将来のプロバイダ支払い)と資産(在庫)が同一単位で一致し、以後のボラはユーザー側に載る。USD建てクレジット(ヘッジ運用が必要)はv2オプション。

### 23.4 バイバックの構造効果と透明性
非MSK建てのAI需要 $X/月 は機械的に $0.9X/月 のMSK買い圧へ変換される(Hyperliquid型フライホイール。手数料5%バーンがこれに重畳する)。信頼装置として**Proof-of-Buyback**を月次公開する: 入金txid集合・約定記録・90/10内訳・在庫残高を、FSLのfactとしてアンカーし誰でも照合可能にする。自社の検証層で自社の会計を証明する構造。

### 23.5 規制設計(弁護士レビュー必須、§12-29)
構造を「ユーザーがMSKを購入する」にすると暗号資産交換業の射程に入り得る。採用する構造は: **ユーザーはサービスクレジットを買う(MSKを受け取らない)**、90%買付は発行者の自己勘定によるトレジャリ方針。クレジットは非譲渡・原則非払戻とし、自家型前払式支払手段(資金決済法)の該当性・届出/供託義務を確認する。入金アドレスのサンクションスクリーニングを運用要件に含める。税務(暗号資産受領収益・期末評価)も同レビューに束ねる。

### 23.6 分散性との整合
ゲートウェイは中央集権的だが、**プロトコル外の交換可能な部品**である: paymasterは誰でも運営でき(§14.3)、90/10は公式ゲートウェイの方針であって合意規則ではない。§21の可視性整合も保たれる(ゲートウェイは支払者を知るが内容を知らない)。v2でブリッジ + MSK/USDC AMMがEVMレーンに成立すれば、90/10分割をコントラクトが強制する**オンチェーンPaymentRouter**へ移行できる — ただしブリッジは業界最大のハック面であり、PQチェーンへの古典アセット持ち込みという新規性も加わるため、監査体制が整うまで着手しない。

### 23.7 Hyperliquid自動執行レール
買付ベニューをHyperliquidに一本化し、全行程を自動化する(§12-30はこれで確定)。
**前提**: MSKのHIP-1スポット上場(USDC/MSKペア)。HIP-1トークンはネイティブMSKの裏付けを要するため、**ネイティブ⇔HLのブリッジ/カストディアン**(閾値署名運用 — Hyperliquidエージェント鍵基盤の転用)が前提プロジェクトになる。上場はHIP-1オークション経由でコストが発生する。
**パイプライン(全自動)**: (1) 搬入 — **SOL/ETHはHyperliquid Unitで原チェーンから直接入金**(Guardianネットワークのlock-and-mint、uSOL/uETHとしてspot板に着地、1〜10分)→ 板でUSDCへ売却。原チェーンDEXスワップは廃止(承認・MEV露出・ガスが消える)。USDCはArbitrumネイティブに集約(Solana USDCはCCTP、Ethereum USDCはブリッジ)してHL正規入金。USDTのみ原チェーンでUSDCへ1回正規化。Unitの最低入金額(例: 0.12 SOL)はユーザー最低チャージ額に反映する。(2) TWAP買付 — バッチ毎スリッページ上限2%、超過分は翌窓へ繰延、fill IDを全件記録。uSOL/uETH→USDCの換金脚はUnit板の既存深度に依存し、ボトルネックは常にMSK/USDC側(§12-34)。(3) バーン執行 — 買付MSKのうちバーン割当(§23.8)を**HLから引き出してネイティブのeaterアドレスへ送付**。wMSKをHL側で焼くのではなく、ネイティブ供給を証明可能に減らす。eaterはプレイメージ不明規約アドレス(`ScriptPublicKey = P2PKH(Hash64("MISAKA-BURN-V1"))`、対応秘密鍵が存在し得ないことを文書化)。(4) 残余はインベントリ補充としてゲートウェイのネイティブアドレスへ。
**障害時**: HL停止・ブリッジ停止中は執行を停止しキューへ積む(強行しない)。ユーザー体験は在庫バッファが吸収し、在庫が閾値を割ったら新規クレジット販売を一時停止する — 裏付けなしのクレジットを売らないことを不変条件とする。

### 23.8 バーン会計恒等式と三層バーン
恒等式: `P = COGS + Burn + Ops`、ここで `Ops = 0.1P` 固定、`COGS + Burn = 0.9P`。COGSはプロバイダ支払い原資であり**焼却不能** — 「90%を全額買ってバーン」は、買付90%は文字通り実行し、バーンは買付分のうち余剰マージン(b×COGS相当)に適用する、という形でのみ成立する。バーンの実量は運営が公開設定するbが決める。

| 層 | 原資 | 機構 | HL依存 |
|---|---|---|---|
| B1 自然燃焼 | 在庫MSKの消費 | 既存の手数料5%バーン(88/5/4/3) | なし(ネイティブ) |
| B2 マージン全焼 | b×COGS(買付余剰) | §23.7(3)の自動eater送付 | あり |
| B3 雑収入 | 失効クレジット・没収残余等 | 四半期毎にまとめてeater送付 | 任意 |

数値例(b = 0.35): 原価$1.00のジョブ → P = $1.50。ops $0.15 / 買付 $1.35 → 在庫 $1.00・**即バーン $0.35**、消費時にB1で更に≈$0.05 → グロスの約26%が焼却。**推奨初期値は b = 0.1〜0.2** — バーンプレミアムは対中央集権APIの価格競争力(§18の生命線)を直接削るため、需要曲線を見ながらダッシュボード公開の上で調整する。最大デフレが目的なら、最安のレバーはHL不要の**ネイティブ手数料バーン率(現行5%)の引き上げ**であり、HLレールは「外部アセット収益をMSK需要+焼却へ変換する」専用装置と役割分担する。MSK直払いにはbが乗らない(B1のみ)ため、MSK保有インセンティブは更に強まる。

### 23.9 受領アセットのルーティングと運営トレジャリ方針
**原則: SOL/ETH/USDTは保有資産ではなく通過資産。** ウォーターフォール:
```
入金 → AMLスクリーニング(不通過は隔離、変換しない)
 ├─ [控除] ガスフロート補充: SOL / ETH 現物を各チェーンのスイープ・Unit入金手数料分のみ保持(v0.11で縮小)
 ├─ 搬入: SOL・ETH → Unit直接入金(§23.7-(1))、USDC → Arbitrum集約、USDT → USDCへ正規化
 ├─ 90%: HL入金 → TWAP買付 → COGS在庫 + バーン(§23.7)
 └─ 10%: 運営。USDC建て保持が原則、使途は次の順:
      (1) HIP-1オークション費 + MSK/USDC板のUSDC側流動性シード(MSK側はトレジャリ)
          — 板厚 = 90%レール自身の執行品質。§12-30残タスクの解
      (2) 監査・法務(§12-29 / §12-32)
      (3) Tier1シードノード(H100 CC)— §22.3cのTEEアンカー審判供給をハード前提から
          運営責務へ引き上げる
      (4) JPY換金は実費支払い時のみ
```
現物フロートを残す唯一の理由は運用ガス(スイープ・rent・スワップ実行)。それを超えるボラ資産の保持は運営の投機であり行わない。クレジットはMSK建て執行価格確定(§23.3)なので、運営の価格露出は「受領 → 正規化」(分オーダー)と「TWAP窓のUSDC/MSK」(≤24h)に閉じる。
**禁止リスト**: SOL/ETHの投機的保有 / フロートの利回り運用(顧客隣接資金でスマートコントラクトリスクを取らない) / **10%からのMSK購入**(90/10の帳簿分離とProof-of-Buyback検算性を守る) / 逆方向換金(§23.1凍結条項)。月次アテステーションにフロート残高・流動性ポジションを含め、ここまでを検算可能にする。

### 23.10 執行レールのリスク
**単一ベニュー**: HL障害＝買付停止(§23.7の縮退で許容)。**ブリッジ**: MSK⇔HLカストディは本設計最大の新規攻撃面 — 上限額・引出遅延・閾値署名者の分散を初期から設計し、TVL段階解放(§9)と同じ規律を課す。**上場実務**: HIP-1オークション費用と維持要件。**法的見え方**: 自社トークンの計画的買付は、執行規律(TWAP・事前公開ルール・裁量排除)とProof-of-Buyback(fill ID + 引出tx + eater残高で誰でも検算可能)を**相場操縦疑義への一次防御**として制度化し、§12-29の弁護士レビューに買付行為の整理を追加する。
