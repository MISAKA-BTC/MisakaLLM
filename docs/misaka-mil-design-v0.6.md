## 21. Tier2匿名性アーキテクチャ — 内容秘匿から連結不能性へ

### 21.1 到達可能な保証の定義
まず既に成立している事実を確定する: E2EEチャネル（§3.2）はプロバイダの推論プロセスで終端するため、**プロバイダとリクエスタ以外の全観測者 — ネットワーク・リレー・ゲートウェイ・チェーン — には内容が見えない**。これはTier1/Tier2共通で、追加実装なしに成立している。

残る問題は「計算する者（プロバイダ）から内容を隠せるか」であり、これに対する数学・工学の答えは明確である: 隠せるのはFHE（10^3〜10^6倍遅延、§3.7）・MPC（§21.5）・ハードウェアTEE（Tier1）のみ。CC非対応のRTX級GPUでは**内容の秘匿は原理的に諦め、保証目標を反転させる**:

> **Tier2の保証目標**: プロバイダは内容（WHAT）を見るが、それが誰のものか（WHO）を、プロバイダを含む誰も特定・連結できない。

内容と身元の結合こそが実害（プロファイリング・恐喝・検閲・規制照会）の源泉であり、切断された内容単体の価値は大きく下がる。切断すべきリンクは3本 — ネットワーク（IP）、支払い（チェーン上の資金経路）、コンテンツ自身（文面内の自己開示）。

### 21.2 ネットワーク切断 — 2ホップリレーのTier2標準化
Tier2では2ホップリレーを標準経路にする（Tier1は任意）: `Requester → R1 → R2 → Provider`。R1はリクエスタIPを知るが宛先プロバイダと内容を知らず、R2はプロバイダを知るがリクエスタIPを知らない。単一リレーでは「IP ↔ プロバイダ」の対応をリレーが握るため、2ホップが最小構成。セルは4KB固定パディング + 送信タイミングのジッタ。

リレーはプロバイダ・第三者が兼務でき（帯域はKB/s級で負担軽微、§16.3）、リレー集合はステーク登録制、選択はリクエスタSDKがステーク加重ランダムに2つ抽選する。報酬はセッション予算からper-MB課金（微額）。レイテンシ予算: +2×域内RTT ≈ **+50〜150ms**で、TTFT目標800msの内側に収まる（§13.1の目標値はリレー込みで維持）。

### 21.3 支払い切断 — 段階ラダー
透明UTXO/EVM上では、セッションエスクローのopen txがリクエスタの資金経路を晒す。切断手段を弱→強の4段で定義する:

| 段 | 手段 | 連結不能性 | 状態 |
|---|---|---|---|
| L0 | セッション毎freshアドレス（§15.2） | 弱（資金グラフ解析で追跡可能） | v1既定 |
| L1 | paymaster/ゲートウェイ集約（§14.3） | 中〜強（プロバイダから見える支払主体はアプリ。k-匿名性 = アプリの利用者数。ゲートウェイは身元を知るが内容は知らない — 内容と身元が同一主体に揃わない） | v1 |
| L2 | **STARKシールド・プリペイドプール**: EVMレーン契約へ固定額面（100 / 1k / 10k MSK）のnote commitmentをdeposit → 消費時にMerkle包含 + nullifierの**STARK証明**を提出し、どのdepositか明かさずセッションエスクローを直接openする | 強（匿名集合 = プール内の同額面deposit全体） | v2 |
| L3 | PQブラインド署名ecash（格子ベースblind signature） | 強 + オフチェーン即時性 | 研究段階（§12-22） |

**PQ整合の選択理由を明記する**: Tornado系のGroth16/BN254ペアリングは非PQのため不採用。STARKはハッシュベースでPQ前提と思想一致し、MISAKAで採用可能な唯一の実用級シールド技術。Cashu等の古典ブラインド署名（DLOGベース）も不採用 — ミントのログが残る限り、将来のCRQCが盲化を破って購入と消費を**遡及的に連結**できる（harvest-now-link-later）。チェーン本体をPQ化した思想と矛盾する妥協はここでもしない。L2の課題はSTARK検証のガス/mass実測と証明生成時間（クライアント側数秒）で、§12-21に送る。

### 21.4 コンテンツ側衛生
文面内の自己開示は暗号では守れないため、SDKに3点を実装する。(a) **ローカルPIIスクラバ**: 送信前に氏名・住所・口座・鍵らしき文字列をローカル検出し警告（検出処理自体を外に出さない）。(b) **private mode**: sticky session無効・会話毎（最強設定では発話毎）にプロバイダをローテーション。KVキャッシュ再利用を失うためTTFT悪化（§13.5とのトレードオフ）をUIで明示し、ユーザーが選ぶ。(c) **会話分散**: 既定でも「会話単位でsticky、会話をまたいだら別プロバイダ」とし、単一プロバイダに同一人物の長期プロファイルが溜まらないようにする。任意のデコイ比率（実プロンプトn件あたりcanary様ダミーm件、費用n+m倍）は上級者向けオプションに留める。

### 21.5 却下した代替案（理由の記録）
**レイヤ分割MPC**（プロバイダ間でモデル層を分担し活性化ベクトルだけ受け渡す）: 第1層担当が生トークンを見る上、中間活性化からの**embedding inversion攻撃で原文が高精度復元できる**ことが知られており、秘匿にならない。非結託仮定と帯域（活性化はトークン毎に数十KB級）も成立しない。**プロンプト分割**（複数プロバイダに断片を送る）: LLM推論は全文脈を要するため意味論的に不成立。**FHE**: §3.7の通り非実用。よってTier2の正解は「隠す」ではなく「切る」である。

### 21.6 Tier2-C（任意）: CPU-TEEニッチ
GPUのCCは不要でも、**CPU TEE（Intel TDX）単体でllama.cppを走らせる**構成なら、MIL-Core 8B Q4で概ね5〜20 tok/s（AMX/AVX-512級サーバCPU）が出る。遅く安い「機密・低速」枠としてTier2-Cを予約する: attestation・E2EE・no-retentionはTier1と同一の測定対象構成、GPUを一切使わない。RTX保有者向けではないが、「TEEはCPUにしかない」事業者の参入枠と、Tier1未満の価格で秘匿が要るユースケースを拾う。

### 21.7 可視性マトリクス（誰に何が見えるか）
| 観測者 | 内容 | リクエスタIP | 支払い主体 |
|---|---|---|---|
| プロバイダ（Tier1） | ✗（TEE） | ✗（リレー時）/ ○（直結時） | L0: △ / L1〜L2: ✗ |
| プロバイダ（Tier2） | **○（仕様として明示）** | ✗（リレー標準） | L0: △ / L1〜L2: ✗ |
| リレーR1 / R2 | ✗（E2EE） | R1: ○ / R2: ✗ | ✗ |
| ゲートウェイ（利用時） | ✗（E2EE素通し） | ○ | ○（ただし内容と揃わない） |
| チェーン・公衆 | ✗（Hash64コミットメントのみ） | ✗ | L0: ○ / L2: ✗ |

設計の核はこの表の**どの行にも「内容○かつ身元○」が現れない**ことにある（Tier2プロバイダ行は内容○・身元✗、ゲートウェイ行は身元○・内容✗）。

## 22. 敵対的プロバイダ設計 — 品質担保・不正検出・不払いメカニズム

### 22.1 原則
**品質担保の還元**: 「回答の質が下がらないこと」を主観評価の問題にしない。カノニカルモデルの良し悪しは§19（DAO 3ゲート）が担い、本節が担保するのは**忠実性** — 登録どおりのMIL-Coreを登録どおりのプロファイルで実行したか — である。忠実性が成立すれば、Tier2の回答品質は定義上MIL-Core自身の品質に一致する。判定はすべて客観（完全一致・署名・物理量）に落とし、主観信号（ユーザー評価）はマッチング優先度にのみ使い、スラッシュには決して使わない。

**経済原則**: 不正は検出可否でなく**期待値で殺す**。不正の期待利得 < 発覚時の総損失、を全パラメータで保証する（§22.6の較正式）。

### 22.2 攻撃カタログ
| # | 手口 | 階層 | 検出 | 帰結 |
|---|---|---|---|---|
| A1 | モデル置換・量子化格下げ（1BやQ2で応答） | Tier2 | canary完全一致（§22.3a）・紛争再実行 | 50%スラッシュ + claim没収 + 被害補償 |
| A2 | 出力切詰め・水増し課金 | 両方 | クライアントが受信トークンを自ら計数、最終レシートと照合 | claim拒否・紛争 |
| A3 | 偽/失効attestation・鍵流用 | Tier1 | 登録nonce束縛・attestation epoch更新義務・ベンダ失効リスト連動 | 登録凍結、偽造立証で100% |
| A4 | 物理限界超過の請求（1台のH100で不可能なトークン流量） | 両方 | **デバイスクラス毎スループット上限**をclaim検証に組込み | 超過分claim自動拒否 + 監査 |
| A5 | レシートequivocation（同一kに矛盾する累積値を二重署名） | 両方 | 矛盾レシート2通の提出 = 完結したfraud proof | 100%スラッシュ（即時・裁量なし） |
| A6 | canary選別（監査だけ真面目に処理） | Tier2 | canary不可識別化（§22.3a） | 選別不能化（予防） |
| A7 | 下請け転送（他プロバイダ/外部APIへ横流し） | Tier2 | 同一プロファイル転送は出力同一のため**原理的に検出不能** | §22.7残余リスク（Tier1は構造的に不可能） |
| A8 | 選択的拒否・特定話題の劣化 | 両方 | 拒否率z-score・canary話題多様性 | レピュテーション降格・マッチング除外 |
| A9 | 紛争委員会との結託 | Tier2 | 一次オラクルをTEEに変更（§22.3c）し人間委員会を縮退 | 攻撃面自体を削除 |
| A10 | 焼き畑Sybil（スラッシュ後に新規登録で復帰） | 両方 | ステーク = 身元コスト・新規probation・Tier1はデバイス証明書ブラックリストが**身元を跨いで残存** | §22.5 |
| A11 | 会話データの持出し・転売 | Tier2 | 立証ほぼ不能 | §21連結不能性が被害を無価値化（予防のみ、§22.7） |

リクエスタ側不正（踏み倒し・偽紛争）は先払いエスクロー（§5.6）とchallengerボンド没収で既に閉じている。

### 22.3 検出機構
**(a) canaryの不可識別化 — §21が監査の前提条件になる**: canaryが見分けられた瞬間、Tier2の全監査は無力化する（A6）。よってcanaryは (1) 通常のリレー/ゲートウェイ経路から到着し、(2) 支払いは§21.3のL1/L2切断経路を通り資金源（Bootstrap Fund）が判別不能で、(3) 内容はVRFシード→**MIL-Core自身**が生成する自然文プロンプト（生成器 = 本番モデルなので文体分布が本番トラフィックと一致し、検証者はシードから同一プロンプトを再現可能）、(4) ポアソン到着・多ターン混在。匿名性インフラは倫理装置であると同時に**監査装置**である — この二重役割を設計上の要件に格上げする。

**(b) 決定論的完全一致**: Tier2プロファイルはシード込みで完全決定論（sampling時もseed固定）。canary・紛争とも判定は出力トークンID列の完全一致で、閾値裁量が存在しない。クロスGPU再現性（§12-3）が崩れた場合のフォールバックは(c)。

**(c) TEEアンカー検証 — Tier1がTier2の審判になる**: 紛争・canary照合の一次オラクルを「任意のTier1 enclaveがTier2決定論プロファイルを再実行し、attested参照出力を返す」に置く。効果は3つ。(1) 人間委員会の結託面（A9）が消える — 信頼根拠はハードウェア測定値。(2) **紛争のプライバシー中立化** — リクエスタは証拠（prompt・seed・出力）をenclaveにのみ開示し、外に出るのは判定とハッシュだけ。Tier2の紛争が人間に内容を晒さない。(3) 異種GPU間の再現性問題を回避（参照系を単一の測定済み構成に固定）。Tier1が存在しない期間のみVRF委員会（§4.2）へフォールバック。検証報酬は紛争ボンド/Fundから支払う。

**(d) 物理スループット上限**: claim検証時に `請求トークン数/epoch ≤ デバイスクラス上限 × 台数` を強制（A4）。attestedデバイス種別（Tier1）またはcanary実測プロファイル（Tier2）が上限の根拠。

**(e) 統計監視と適応監査**: プロバイダ毎に canary一致率（100%以外は即異常）・拒否/短応答率z-score・レイテンシとハードウェア主張の整合・出力長分布ドリフトを常時計測。疑義スコア上昇でそのプロバイダへのcanary混入率ρを自動増加（適応監査 — 全体コストを上げずに疑わしい先へ監査を集中）。

### 22.4 支払い機構 — 不正に報酬を渡さない
**Tier1: 即時claim**。enclave署名レシートは実行証明そのものであり、チャレンジ窓なし（W=0）。
**Tier2: 楽観claim + チャレンジ窓W（24h）**。claimはW経過後に確定。窓内に紛争が開けば当該claimは凍結され、TEE再実行の判定まで支払われない。プロバイダの拘束は「直近24h分の売上」に留まり運転資金負担は軽微。

| 事由 | スラッシュ | 追加帰結 |
|---|---|---|
| A5 equivocation・偽attestation立証 | **100%** | 永久追放（Tier1はデバイス単位でも） |
| A1 出力不一致（紛争敗訴） | **50%** | 当該セッション全claim没収、被害リクエスタへ返金 |
| A4 物理限界超過 | 超過claim没収 + 20% | 監査強化 |
| canary不一致（単発） | 0（スラッシュなし） | probation化・ρ増・マッチング降格 — 過失と攻撃を段階で区別 |
| タイムアウト・失踪 | 軽微（§5.5） | escrow全額refund |

スラッシュ分配はPoS-v2の4-way（reporter 10 / **victim 40** / reserve 40 / burn 10）を転用し、被害リクエスタの補償原資を没収ステークから出す。§5.5の表は本表で拡張される。

### 22.5 評判・新規参入の経済
レピュテーションq = canary成績・紛争履歴・SLA実測の合成（主観評価は含めない）。新規身元はprobation: 初期14日はエスクロー露出上限（例: 同時1k MSK）・マッチング優先度最下位・ρ高め。これにより焼き畑Sybil（A10）の再登録コスト = 失ったステーク + probation期間の機会損失 + （Tier1では）ブラックリスト済みデバイスの買い替え、となり§22.6の式に組み込まれる。

### 22.6 経済較正 — 不正のEVを負にする式
置換系不正の1ジョブあたり利得を g（≒ 手数料に占める計算コストの節約分）、canary混入率をρ、canary検出力を完全一致前提で≈1とすると、発覚までの期待ジョブ数は 1/ρ、不正の期待総利得は g/ρ。発覚時の損失はステークS + 将来収益の現在価値V + 未確定claim。安全条件:
```
g / ρ  <  S + V        ⇔        ρ  >  g / (S + V)
```
数値例: 平均手数料1 MSK/job、置換節約 g = 0.5 MSK、S = 100k MSK（クラスB）、V ≈ 50k MSK とすると必要ρ ≈ 0.0003%。**基準ρ = 1%は安全係数〜3,000倍**であり、逆に言えば委任ステーキング等でSを下げる圧力への定量的な下限を与える。チャレンジ窓Wは「発覚前に引き出せる不正収益」を直近W時間分に制限する第2の抑え。この式（と適応ρ）を§10のパラメータ結合として明文化し、S・ρ・Wの独立な変更を禁じる。

### 22.7 残余リスクの正直表
**A7 下請け転送**は、同一プロファイルへの転送なら出力が同一のため原理的に検出できない。品質は無害（同じ答え）だが、Tier2の可視範囲が契約相手以外へ広がる — 対策は§21の連結不能性（広がった先でも身元と結合しない）とToS上の禁止に留まることを明記する。**A11 データ持出し**も立証不能であり、防御は同じく「持ち出しても誰のものか分からない」構造に置く。**決定論の崩壊**（§12-3が不成立）時は判定系を全面的にTEEアンカー（§22.3c）へ寄せ、Tier2の紛争がTier1供給に依存する体制になる — Tier1ゼロ地帯でのTier2単独運用は監査力が落ちることを運用制約として認める。TEEベンダ侵害は§3.6・§9の既存限界に帰着する。
