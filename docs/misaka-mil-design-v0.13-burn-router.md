## §24.7 逆周期バーンルーターとProvider Stabilization Pool（PSP）
**目的**: ネットワーク収益が大きい局面ではバーン（デフレ）を最大化し、小さい局面ではバーン予定フローを稼働プロバイダへ再配分して床を厚くする。一行で: **買い圧は無条件、バーンは好況連動、プロバイダ支援は不況連動。**

**対象フロー**: B1（ネイティブ手数料5%バーン）と B2（ゲートウェイのバーンマージン買付分）。B2の**買付自体は無条件に継続**し、切り替わるのは宛先（eater ⇔ PSP）のみ — MSK買い圧は景気に依存しない。両フローは単一のルーター契約（EVMレーン）を経由し、Proof-of-Buyback はルーター tx を参照する。

**モード判定（連続ランプ、二値にしない）**:
```
I = 直近7日の手数料収入（FSLレートUSD換算） ÷ 非standbyのattested稼働デバイス数
s = clamp( (I − I_low) / (I_high − I_low), 0, 1 )
バーンへ: s × (B1 + B2)     PSPへ: (1 − s) × (B1 + B2)
```
二値閾値は境界でのフラッピングと崖ゲーミングを招くため、ヒステリシス的に働く線形ランプとバンド [I_low, I_high]（ガバナンス設定、初期較正は §12-38）を用いる。

**PSPの分配**: エポック毎に `PSP_i = PSP残高 × min(served_tokens_i / Σserved_tokens, 5%)` — **bond比例ではなく検証済み提供トークン数比例**（レシート由来）。この支援は手数料側の補填なので「発行は在席、手数料は稼働」（§24.2）の軸を守り、不況でも実際に稼働した者にのみ実効単価の上乗せとして届く。standby中のデバイスは分配対象外かつ指標Iの分母からも除外。

**ゲーミング分析**: 収入の過少偽装 = 受注拒否で自滅。wash trading での過大偽装 = バーン・スプリット分の実費を必ず失い、しかもバーンモードを強めるだけで攻撃者に利得がない。standby偽装で分母を膨らませる攻撃は分母定義（非standbyのみ）で遮断。残る面は FSL レート操作によるモード反転で、7日移動平均が鈍化させる — 較正・攻撃費用の定量は §12-38。

**§24.5との整合（明文）**: 本機構は発行・キャップ・coinbase分割（70/25/5）に一切触れない。再配分されるのは**ネットワークが実際に稼いだフローのみ**であり、収益ゼロならPSPもゼロ。したがって希釈による死のスパイラル経路は存在せず、「全員黒字の保証」でもない（§24.6 の線引きは不変）。
